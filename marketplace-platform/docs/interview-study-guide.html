Technology Summaries

Quick reference for every technology used in the Marketplace Platform

Java 21 Virtual Threads

Concurrency without complexity

Lightweight threads managed by the JVM (~1KB each vs ~1MB for platform threads). When a virtual thread hits a blocking call (gRPC, DB, Kafka), it parks and frees the carrier thread for other work. Enables thousands of concurrent requests with no code changes.

Key insight: Write normal blocking code, get non-blocking performance. No reactive programming needed.

application.yml:5 spring.threads.virtual.enabled: true

Apache Kafka

Async messaging between services

A distributed message broker. Services drop messages into topics (named mailboxes). Other services pick them up when ready. Messages are stored on disk — if a consumer is down, messages wait and are never lost.

Our topics: order-events (3 partitions) and inventory-events (3 partitions). Uses orderId as key for partition routing.

OrderEventProducer.java InventoryEventConsumer.java KRaft mode (no Zookeeper)

gRPC

Fast service-to-service calls

Binary protocol over HTTP/2 for synchronous calls between services. A .proto file defines the contract — both services generate Java code from it. ~10x smaller payloads than JSON. Used when the caller needs an answer right now.

Our RPCs: CheckStock and ReserveStock, defined in inventory.proto.

inventory.proto InventoryGrpcServer.java InventoryGrpcClient.java

PostgreSQL

Relational database for orders

Stores orders and order items in tables with ACID transactions. Guarantees that an order and all its items are saved together or not at all (atomicity). Uses JPA/Hibernate for ORM with @Transactional.

Tables: orders (UUID, status, total) and order_items (product, qty, price).

Order.java OrderRepository.java @Transactional

MongoDB

Document database for inventory

Stores products as self-contained JSON-like documents. No fixed schema — flexible for varying product attributes. Uses @Version for optimistic locking to handle concurrent stock updates safely.

Collections: products (stock, reserved) and stock_events (audit trail).

Product.java StockEvent.java Optimistic Locking

REST APIs

HTTP endpoints for clients

Standard HTTP/JSON API for external clients (frontend). Proper HTTP methods (POST, GET, PUT), status codes (201, 404, 409, 422), pagination, and Bean Validation (@NotBlank, @Min).

Endpoints: POST/GET /api/orders, PUT /cancel, GET /api/products.

OrderController.java ProductController.java 6 endpoints total

Observability

Metrics, health, logging

Micrometer counters and timers track business metrics. Spring Boot Actuator exposes health checks and Prometheus-format metrics. Structured logging with SLF4J includes contextual data (orderId, productId).

Metrics: orders.created.total, orders.rejected.total, orders.creation.duration.

MetricsConfig.java /actuator/prometheus

Microservices

Architecture principles

Two independently deployable services, each with its own database (polyglot persistence). Communicate via gRPC (sync) and Kafka (async). Independent event models — no shared library. Each service has its own Maven module.

Services: Order Service (port 8080) + Inventory Service (port 8081/9090).

Database per service Loose coupling

Testcontainers

Real infrastructure in tests

Spins up real Docker containers (PostgreSQL, MongoDB, Kafka) for integration tests. No mocks for infrastructure — catches real bugs. Uses @DynamicPropertySource to inject container URLs and Awaitility for async assertions.

OrderControllerTest.java OrderEventConsumerTest.java @Testcontainers

Docker & Infrastructure

Containerized services

Docker Compose runs PostgreSQL 16, MongoDB 7, and Kafka 3.7 (KRaft mode). KRaft eliminates Zookeeper dependency. Three containers, one docker compose up command.

docker-compose.yml KRaft = no Zookeeper

Spec Kit (AI-First Development)

Spec-Driven Development with AI agents

A CLI tool for Spec-Driven Development where specifications become executable artifacts. Define a constitution (non-negotiable principles), then specify what to build and why, plan the tech stack, break into tasks, and let AI agents implement. Prevents context loss, scope creep, and inconsistent decisions across prompts.

Workflow: Init → Constitution → Specify (what+why) → Plan (how) → Tasks → Implement
